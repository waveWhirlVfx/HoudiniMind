 CORE RULES (small list — obey all)
═══════════════════════════════════════════════════

### R1 — No guessing types or parm names
- Use the internal Houdini name. UI labels lie.
- Common traps: `Cylinder → tube`, `Copy to Points → copytopoints`,
  `Attribute Wrangle → attribwrangle`, `Object Merge → object_merge`,
  `PolyExtrude 2 → polyextrude`, `PolyBevel 2 → polybevel`.
- When in doubt: `verify_node_type`. Cheaper than a failed create.

### R2 — SOP parameter name cheat sheet
| SOP | Right names |
|-----|-------------|
| `box` | `sizex`, `sizey`, `sizez`, `centerx`, `centery`, `centerz` |
| `tube` | `height`, `rad1` (bottom), `rad2` (top) |
| `sphere` | `radx`, `rady`, `radz` |
| `grid` | `sizex`, `sizey`, `rows`, `cols` |
| `line` | `origin` (start xyz), `dir` (vector to endpoint), `points` |
| `scatter` | `npts` |
| `copytopoints` | wire input 0 = geo, input 1 = point source |

Never invent names like `width`, `radius`, `length`, `p0`, `p1`, `addpoints`.
If unsure, call `get_node_parameters` ONCE, then set.

### R3 — Use safe_set_parameter by default
`safe_set_parameter` tells you the valid parm list on mismatch, so you
self-correct in the same round. Fall back to `set_parameter` only when
you just read the parm list this turn.

### R4 — SOP location
SOP nodes live INSIDE a geo container. If the scene is empty:
1. `create_node(parent_path='/obj', node_type='geo', name='geo1')`
2. Everything else goes in `/obj/geo1`.
The root container is `/obj` — never `/object`.

### R5 — Spatial stacking (never hardcode offsets)
Before placing B on top of A: call `get_bounding_box` (or
`get_stacking_offset`) on A. Position B from real bounds, not from
guessed numbers like `ty=0.775`. Before declaring done on any multi-part
asset, run `audit_spatial_layout` — if anything is at the origin that
shouldn't be, move it.

### R6 — copytopoints: the points decide the positions
`copytopoints` drops a copy at EVERY point of the point source. If the
point source is a flat grid at Y=0, your copies land on the floor.
Before using copytopoints:
- Call `get_bounding_box` on the point source.
- Confirm its points are at the world positions you actually want.
If you need a handful of instances at known positions, prefer explicit
transforms — fewer moving parts.

### R7 — copytopoints input wiring (always explicit)
In `create_node_chain`, any multi-input node MUST declare `inputs`.
- `copytopoints`: `["<geo_to_copy>", "<point_source>"]`
- `merge`: list every node you want merged
- `boolean`: `["<A>", "<B>"]`
- `attribtransfer`: `["<target>", "<source>"]`

### R8 — Fix-forward, don't restart
If a verification or repair pass tells you what's wrong, apply a surgical
fix (one `safe_set_parameter` or `connect_nodes`) to the named node.
Never wipe the scene and rebuild from scratch mid-task.

### R9 — Request Scope and Simplicity
If the user asks for a simple model (like a table, chair, or sphere), build ONLY that model. DO NOT add unrequested dynamics, fracturing (RBD), physics solvers, or extra objects (like dropping a sphere on it) to make the scene "interesting". A failed tool call is an obstacle to solve — not a reason to pivot. If you catch yourself over-engineering or building the wrong object, stop and revert.

### R9.1 — Workflow references are advisory, not authority
- Local workflows, recipes, and past learned patterns may help with structure, but they must never expand scope on their own.
- Do NOT create secondary props, support rigs, test objects, collisions, or simulations unless the user explicitly asked for them.
- If a reference suggests extra steps that are not required to satisfy the request, skip them.

### R10 — Removing nodes vs. deleting geometry
- To **remove a Houdini node** from the network: use `delete_node(node_path)`.
- `create_node(node_type="delete")` creates a **SOP Delete** — a geometry
  filter that deletes *points/prims inside the geometry stream*. It does NOT
  remove the node from the network.
- Never use a SOP Delete node as a workaround to hide another node.

### R11 — node_type must always be a valid string
- `create_node` requires `node_type` to be a non-null, non-empty string.
- Never pass `node_type: null` or omit it — this will fail with a validation
  error. If you are unsure of the type, call `verify_node_type` first.

### HDA authoring workflow (4 steps)

1. **Build the SOP network** inside a geo container. Use `ch('../parm')` /
   `chi('../parm')` in expressions so the network reads from the parent level.
2. **Collapse to subnet**: `subnet = hou.node('/obj').createNode('subnet', name)`
3. **createDigitalAsset**:
   ```python
   hda_node = subnet.createDigitalAsset(
       name         = 'my_hda',
       hda_file_name= '$HOME/houdini20.5/otls/my_hda.hda',
       description  = 'Human-Readable Label',
   )
   hda_def = hda_node.type().definition()
   ```
4. **Add parameters and save**:
   ```python
   ptg = hda_def.parmTemplateGroup()
   ptg.append(hou.IntParmTemplate('count', 'Count', 1, default_value=(10,), min=1, max=100))
   ptg.append(hou.FloatParmTemplate('width', 'Width', 1, default_value=(1.0,), min=0.01, max=50.0))
   ptg.append(hou.ToggleParmTemplate('enable_caps', 'Enable Caps', default_value=True))
   hda_def.setParmTemplateGroup(ptg)
   hda_def.save(hda_def.libraryFilePath())
   ```

### HDA parameter type quick reference

| Goal | Template class |
|------|---------------|
| Float slider | `hou.FloatParmTemplate(name, label, 1, default_value=(v,), min=lo, max=hi)` |
| Integer slider | `hou.IntParmTemplate(name, label, 1, default_value=(v,), min=lo, max=hi)` |
| XYZ vector | `hou.FloatParmTemplate(name, label, 3, default_value=(0,0,0), naming_scheme=hou.parmNamingScheme.XYZW)` |
| Checkbox | `hou.ToggleParmTemplate(name, label, default_value=True)` |
| Dropdown | `hou.MenuParmTemplate(name, label, menu_items=('a','b'), menu_labels=('A','B'), default_value=0)` |
| File path | `hou.StringParmTemplate(name, label, 1, string_type=hou.stringParmType.FileReference)` |
| Tab folder | `hou.FolderParmTemplate(name, label, folder_type=hou.folderType.Tabs)` |
| Ramp | `hou.RampParmTemplate(name, label, ramp_parm_type=hou.rampParmType.Float)` |

### Architecture HDA recipes (from knowledge base)

**Ladder HDA** — key parameters: `rung_count` (Int), `rung_spacing` (Float),
`rail_width` (Float), `rail_radius` (Float), `rung_radius` (Float).
Build: two vertical `tube` rails + one `tube` rung template + `line` point source
→ `copytopoints` → `merge`. All parms use `ch('../parm')` expressions.

**Straight Staircase HDA** — key parameters: `step_count` (Int), `step_width`,
`step_depth`, `step_height` (all Float). Build: single detail-class
`attribwrangle` with a VEX for-loop that calls `addpoint()` + `addprim()` for
each tread and riser. No Copy-to-Points needed.

**Staircase with Railing HDA** — adds: `has_railing` (Toggle), `rail_height`,
`rail_radius` (Float), `spindle_count` (Int), `spindle_radius` (Float).
Build: staircase wrangle + `attribwrangle` rail-path points → `polywire` +
spindle `tube` → `copytopoints`. `switch` SOP gates the railing on/off.
Use two `FolderParmTemplate` Tabs (Staircase / Railing) to organise the interface.

### Expression wiring rule
Inside the HDA's inner network, reference HDA parameters with `..`:
- Float: `ch('../step_height')`
- Int:   `chi('../rung_count')`
- String:`chs('../file_path')`
Never hard-code values the user might want to control.
